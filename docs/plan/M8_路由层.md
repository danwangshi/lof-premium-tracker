## 二、接口契约

### 2.1 统一响应格式

成功：`{code: 0, data: {...}, meta: {...}, message: "success"}`

错误：`{code: 40102, message: "登录已过期", detail: "..."}`

### 2.2 分页参数

- page：页码，从 1 开始，默认 1
- size：每页条数，默认 50，最大 100

### 2.3 排序参数

- sort：排序字段（白名单校验防 SQL 注入）
- order：asc/desc，默认 desc

### 2.4 日期格式

ISO 8601 带时区：`2026-05-29T14:55:00+08:00`

### 2.5 空值处理

- 字段值为空 → 返回 null，不省略字段
- 查询结果为空 → `{code: 0, data: [], meta: {total: 0}}`

## 三、schemas/request.py

### 3.1 请求模型清单

| 模型 | 用途 | 关键字段 | 校验 |
|------|------|---------|------|
| FundListParams | 基金列表 | page/size/sort(白名单)/order/premium_min/max/amount_min/max/fund_type/purchase_status/turnover_min/volume_ratio_min/filter/search | sort 白名单，size ≤ 100 |
| FundBatchParams | 批量详情 | codes(逗号分隔，最多 50 只) | codes 数量 ≤ 50 |
| DailyQueryParams | 日线查询 | fields(白名单)/from_date/to_date/sort/order/limit(≤1000) | from ≤ to，to ≤ 今天 |
| FormulaCreate | 创建公式 | name(非空)/expression(非空)/description | name 不和字段名冲突 |
| FormulaUpdate | 修改公式 | name/expression/description + version（乐观锁） | If-Match header |
| FormulaGroupCreate | 创建公式组 | name/formula_ids(列表)/filter_expr/sort_by/sort_order | formula_ids 中的 ID 存在 |
| AlertCreate | 创建预警 | fund_code/condition(JSON)/channel/webhook_url | condition 格式校验 |
| RefreshParams | 触发刷新 | type(realtime/nav/kline/all)/sync(默认 false) | type 白名单 |
| AdminConfigUpdate | 热更新配置 | key(白名单)/value | key 在允许列表中 |

### 3.2 排序字段白名单

premium_rate / close / amount / turnover_rate / change_pct / nav / volume / float_share / aum / code / name

排序字段不在白名单中 → 使用默认排序（amount DESC）。在 Pydantic validator 中校验。

### 3.3 基金代码清洗

clean_fund_code 函数（Pydantic validator 或公共依赖）：
1. strip 空格
2. 全角转半角（１６０６４４ → 160644）
3. 去掉非数字字符（逗号、连字符等）
4. zfill(6) 补零

所有接受基金代码的端点（路径参数和查询参数）都经过这个清洗函数。

### 3.4 日期参数校验

DailyQueryParams 的校验逻辑：
- from_date > to_date → 自动交换（不报错）
- to_date > 今天 → 自动截断为今天
- 两者都不传 → 默认最近 60 天

### 3.5 字段白名单校验

DailyQueryParams 的 fields 参数：
- 请求中不在白名单的字段 → 过滤掉（静默忽略）
- 必须包含 trade_date（自动添加）
- 响应 meta 中返回 ignored_fields 列表，告知前端哪些字段被过滤

## 六、Cache-Control 设置

| 端点 | Cache-Control | 理由 |
|------|---------------|------|
| /funds（列表） | max-age=30 | 交易时段每 5 分钟更新 |
| /funds/{code}（详情） | max-age=300 | 不需要太实时 |
| /funds/{code}/chart | max-age=3600 | 历史数据不变 |
| /assets | max-age=3600 | 每周更新 |
| /formulas | no-cache | 用户私有数据 |
| /health | no-store | 不缓存 |

Cloudflare 边缘节点可以直接返回缓存的响应，减少服务器负载。历史图表数据尤其受益。

## 七、CSV 导出

复用 fund_service 的查询逻辑（不在路由层重复写筛选逻辑）。

CSV 格式标准：RFC 4180。所有字符串字段用双引号包裹，内部双引号转义为两个双引号。Python 标准库 csv.writer 默认处理。

文件名自动命名：包含筛选条件摘要。
- 示例：funds_etf_premium_gt5_20260529.csv
- 示例：fund_160644_daily_20260101_20260529.csv

响应头：Content-Disposition: attachment; filename=...

## 八、管理后台页面

Nginx 静态服务 /admin/，不经过 FastAPI。

前端技术：纯 HTML + CSS + JS，不引入框架。数据通过 fetch 调用 /api/v1/admin/* API 获取。

鉴权：页面加载时从 localStorage 读 Supabase token，调 /api/v1/health 验证。token 无效或不是 admin → 跳转主应用登录页。

页面区块：系统状态 / 数据新鲜度 / 采集状态 / 数据库概览 / 操作按钮 / 日志查看 / 配置热更新 / 审计日志。

## 九、路由注册

routers/__init__.py 中创建一个主 APIRouter，注册所有子路由：

```
主路由 prefix="/api/v1"
  ├── system.router
  ├── funds.router
  ├── assets.router
  ├── data.router
  ├── formulas.router
  ├── watchlist.router
  ├── alerts.router
  ├── admin.router (prefix="/api/v1/admin")
  └── stream.router
```

app.py 中 app.include_router(main_router)。
