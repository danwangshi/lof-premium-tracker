# M5 公式引擎 详细设计

## 一、架构

后端 Python（权威定义 + 存储 + 校验 + P1 付费计算）
前端 TypeScript（主体求值引擎 + 免费用户本地计算）

核心约束：
- 后端不存储任何计算结果
- 结果只存在于用户本地（浏览器 localStorage / 内存）
- 后端只负责：公式定义的 CRUD、表达式校验、P1 付费计算（返回后不存储）

计算通道：
- 免费通道：前端 JS 引擎本地计算（用户设备 CPU，零服务器成本）
- 付费通道：后端 Python 计算（P1，返回结果后不存储，函数返回即释放内存）

## 四、parser.py

### 4.1 函数清单

| 函数 | 输入 | 输出 | 说明 |
|------|------|------|------|
| parse(expression) | 字符串 | AST | 语法错误抛 40002 |
| validate(tree) | AST | None | 递归校验安全性，非法节点抛 40003 |
| complexity(tree) | AST | int（节点数） | 超 100 抛 40004 |
| analyze_dependencies(formulas) | 公式列表 | 拓扑排序列表 | 依赖分析 + 环检测 |
| validate_and_compile(expression) | 字符串 | AST | parse → validate → complexity 组合调用 |

### 4.2 parse 函数

调用 Python 内置 ast.parse(expression, mode='eval')。语法错误时捕获 SyntaxError，转为 ValidationException(40002)。

LRU 缓存：lru_cache(maxsize=256)。相同 expression 字符串直接返回已解析的 AST，不重复解析。

### 4.3 validate 函数（递归校验）

逐节点类型校验，只允许白名单内的节点和操作：

| 节点类型 | 允许条件 | 不允许则抛 |
|----------|----------|-----------|
| BinOp（二元运算） | 操作符 ∈ {+, -, *, /, %, **} | 40003 |
| UnaryOp（一元运算） | 操作符 ∈ {-}（取负） | 40003 |
| Compare（比较） | 操作符 ∈ {>, <, >=, <=, ==, !=} | 40003 |
| IfExp（三元表达式） | test/body/orelse 都递归校验 | 40003 |
| Call（函数调用） | 函数名在安全函数白名单 + 参数数量校验 | 40003 |
| Constant（常量） | 只允许 int/float 类型 | 40003 |
| Name（变量名） | 必须在 ALLOWED_FIELD_IDS 内 | 40003 |
| 其他所有类型 | 直接拒绝（Attribute, List, Lambda, Import 等） | 40003 |

### 4.4 安全函数白名单

| 函数 | 参数要求 | Python 实现 | 说明 |
|------|----------|------------|------|
| abs(x) | 1 个参数 | abs | 绝对值 |
| max(a, b, ...) | ≥ 2 个参数 | max | 最大值 |
| min(a, b, ...) | ≥ 2 个参数 | min | 最小值 |
| round(x, n) | 1-2 个参数 | round | 四舍五入 |
| ifnone(x, default) | 2 个参数 | 自定义 | x 为 None 时返回 default |

ifnone 是自定义函数（不是 Python 内置），evaluator 中实现：如果第一个参数为 None，返回第二个参数。

### 4.5 complexity 函数

递归遍历 AST，计数节点总数。超过 100 个节点 → 抛 ValidationException(40004, "表达式过于复杂", detail={nodes: N, max: 100})。

### 4.6 analyze_dependencies 函数（公式间引用）

输入：公式组中的所有公式 [{name, expression}, ...]

处理逻辑：
1. 对每个公式调用 parse + validate，得到 AST
2. 提取每个 AST 中的所有 Name 节点（引用的变量名）
3. 区分：Name ∈ ALLOWED_FIELD_IDS → 内置字段；Name ∈ 其他公式名称 → 公式间引用
4. 构建有向图：公式 X 引用公式 Y → 边 X → Y
5. DFS 检测环：从每个节点出发做深度优先搜索，如果回到已访问节点则存在循环
6. 有环 → 抛 ValidationException(40002, "公式存在循环依赖", detail={cycle: [...路径]})
7. 无环 → 返回拓扑排序列表（求值顺序）

### 4.7 命名冲突规则

- 公式名称不能和 ALLOWED_FIELD_IDS 重叠（如不能命名为 "close" 或 "nav"）
- 同一用户内公式名称唯一（不同用户可以重名）
- 校验时机：创建/修改公式时

### 4.8 允许的 AST 节点类型

不允许的节点类型（直接拒绝）：
- Attribute（属性访问，如 obj.method）
- List / Tuple / Set / Dict（集合类型）
- ListComp / GeneratorExp（推导式）
- Lambda
- Import
- 任何其他非白名单节点

## 五、evaluator.py

### 5.1 函数清单

| 函数 | 输入 | 输出 | 说明 |
|------|------|------|------|
| compile_formula(tree) | AST | 求值闭包 | AST 编译为 fn(context) → float/None |
| evaluate(fn, context) | 闭包 + 数据字典 | float/bool/None | 单次求值 |
| batch_evaluate(formulas, funds_data) | 公式列表 + 基金数据 | {code: {name: value}} | 批量求值 |

### 5.2 compile_formula 函数

将 AST 编译为一个轻量闭包函数 fn(context) → float | None。

闭包内部已经确定了 AST 的结构和操作类型，不需要每次调用时重新判断节点类型。对 1,500 只基金调用同一个 fn，比每次重新遍历 AST 快 3-5 倍。

### 5.3 evaluate 函数

调用编译后的闭包。context 是字典，包含基金基础数据（如 {close: 2.323, nav: 2.048, premium_rate: 13.43}）和前置公式的计算结果。

容错规则：
- 字段值为 None → 返回 None（不报错）
- 除零 → 返回 None（不报错）
- 任何其他异常 → 返回 None（不报错）
- 比较运算返回 bool（Python bool 是 int 子类型，可参与算术运算）

返回值类型：float | bool | None。类型提示应为 Union[float, bool, None]。

### 5.4 batch_evaluate 函数

输入：
- formulas：公式组定义 [{name, expression}, ...]
- funds_data：基金基础数据 [{code, name, close, nav, ...}, ...]

处理流程：
1. 调用 analyze_dependencies(formulas) → 得到拓扑排序 + 环检测
2. 按拓扑排序逐公式调用 validate_and_compile → 得到求值闭包列表
3. 性能预估：基金数 × 公式数 × 平均节点数 × 单节点耗时。超 5 秒 → 拒绝执行
4. 对每只基金，按拓扑排序顺序逐公式求值：
   - 将基金基础数据作为初始 context
   - 求值公式 A → 结果存入 context[公式A的名称]
   - 求值公式 B（context 中已有 A 的值）→ 存入 context[公式B的名称]
   - 以此类推
5. 输出：{fund_code: {formula_name: result, ...}, ...}

每只基金独立处理，一个基金的异常不影响其他基金（异常的基金 result 为 None）。

### 5.5 性能边界

| 限制项 | 阈值 | 超限处理 |
|--------|------|----------|
| 基金数量 | 2,000 | 拒绝，提示筛选范围 |
| 公式数量 | 10 | 拒绝，提示精简公式 |
| 单公式 AST 节点数 | 100 | 拒绝（在 validate 阶段） |
| 预估耗时 | 5 秒 | 拒绝，提示减少基金或公式 |

预估公式：基金数 × 公式数 × 平均节点数 × 0.001ms（单节点耗时经验值）。

### 5.6 函数返回后内存释放

batch_evaluate 返回结果后，所有中间变量（AST、闭包、context）随函数栈帧释放。后端不保留任何计算结果。

## 六、前端公式引擎

### 6.1 fields.ts

从构建时生成的 fields.json 文件导入。不在前端手写字段定义，保证前后端一致。

### 6.2 parser.ts

JS 没有内置 ast 模块，需要自己实现解析器或使用第三方库。

推荐方案：用 expr-eval 库做底层表达式解析（成熟、轻量、支持三元表达式），在上层加白名单校验层（检查变量名是否在 ALLOWED_FIELD_IDS 内）。

校验逻辑和 Python parser.py 完全一致：
- 变量名白名单校验
- 函数名白名单校验
- 节点数上限校验

### 6.3 evaluator.ts

和 Python evaluator.py 同逻辑的 TypeScript 实现。

关键差异：
- 容错返回值统一 null（不是 undefined、NaN、Infinity）
- ifnone 函数的实现：`x === null ? default : x`
- 批量求值时用 for 循环而非 map（性能更好）

### 6.4 前后端一致性保证

| 机制 | 说明 |
|------|------|
| 字段白名单共享 | fields.py 导出 JSON → fields.ts 导入，单一来源 |
| 安全函数白名单共享 | 同上，导出为 JSON |
| 校验逻辑一致性 | 两边实现同一套规则 |
| 测试用例共享 | test_cases.json，Python 和 TypeScript 各跑一遍，结果必须一致 |

### 6.5 test_cases.json 格式

每个测试用例包含：expression（表达式）、valid（是否合法）、context（测试数据）、expected（预期结果，null 表示 None）、error_code（预期错误码）。

Python 测试和 TypeScript 测试读取同一份 JSON，分别执行，对比结果。浮点数比较用精度容差（如 0.0001）。

## 七、公式间引用详解

### 7.1 场景

用户创建多个公式，公式之间可以引用彼此的计算结果：

公式 A（nav_premium）：close / nav - 1
公式 B（adjusted）：nav_premium * 2 + change_pct
公式 C（signal）：adjusted > 0.05 ? 1 : 0

引用链：C → B → A → 内置字段。

### 7.2 求值顺序

analyze_dependencies 返回拓扑排序 [A, B, C]。求值时：

1. 用基金基础数据作为 context
2. 求值 A → 结果存入 context["nav_premium"]
3. 求值 B（context 中已有 nav_premium）→ 存入 context["adjusted"]
4. 求值 C（context 中已有 adjusted）→ 存入 context["signal"]
5. 输出 {code, nav_premium: ..., adjusted: ..., signal: ...}

所有中间变量和最终结果全部保留，前端可以展示任意中间变量。

### 7.3 循环依赖检测

如果公式 A 引用 C、C 引用 B、B 引用 A → 形成环 A → C → B → A。

检测方式：DFS 遍历有向图，维护访问状态（未访问/访问中/已完成）。遇到"访问中"的节点 → 发现环。

错误响应：{code: 40002, message: "公式存在循环依赖", detail: {cycle: ["A", "C", "B", "A"]}}

## 九、注意事项

### 安全

- 绝对不用 Python eval() 执行用户表达式
- AST 白名单校验是唯一的准入机制
- 前端 JS 引擎同样需要白名单校验（不能信任前端输入）

### 性能

- LRU 缓存避免重复解析同一表达式
- compile_formula 生成闭包，1500 基金调用同一闭包而非每次遍历 AST
- 批量求值时按拓扑序编译一次，对所有基金复用
- 性能边界在 batch_evaluate 入口处检查，超限立即拒绝

### 类型一致性

- Python 返回 None 的地方，JS 必须返回 null（不是 undefined）
- 比较运算返回 bool，bool 参与算术运算时 Python 中 True=1, False=0
- 前端公式编辑器中提示用户：比较运算的结果是布尔值，直接参与算术可能产生非预期结果

### 公式编辑器支撑

- 字段名自动补全：输入 "pre" 弹出 premium_rate / premium_3d
- 语法高亮：字段名绿色、运算符灰色、函数名蓝色、数字橙色
- 错误标记：校验失败时在错误位置加红色下划线
- 常用示例：点击填入预设表达式
- 实时预览：输入公式后用前 5 只基金计算并展示结果
