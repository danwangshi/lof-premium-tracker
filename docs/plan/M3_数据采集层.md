# M3 数据采集层 详细设计

## 一、模块总览

| 文件 | 主源 | 备源 | 事件 type | 说明 |
|------|------|------|-----------|------|
| fetchers/__init__.py | - | - | - | 包初始化 + 工具函数 + 统一导出 |
| fetchers/realtime.py | push2 clist | 腾讯 qt | realtime | 实时行情，单次批量 |
| fetchers/fundamental.py | lsjz API | - | nav | 净值+申赎+费率 |
| fetchers/historical.py | push2his | AkShare | kline | 日线 K 线（基金+资产） |
| fetchers/info.py | fundf10 | - | info | 持仓+基础信息（分批爬取） |

## 二、共同约束

### 2.1 职责边界

fetcher 只负责：
- 从外部 API 获取原始数据
- 做最基础的格式清洗（空值替换、代码补零）
- 发布 Stream 事件

fetcher 不负责：
- 字段映射到统一格式（归 M4 normalize）
- 派生字段计算（归 M4 calculator）
- 写入 DB 或 Redis 缓存（归 M4 pipeline/saver）

### 2.2 输出格式

fetcher 输出原始字段名（push2 的 f2/f3/f12 等），不做映射。M4 normalize 负责将不同来源的原始字段映射到统一格式。

fetch_source 字段标记数据来源（"push2" / "tencent" / "lsjz" / "push2his" / "akshare" / "fundf10"），透传到 DB 的 data_source 字段。

### 2.3 异常处理

- 网络超时/连接失败 → 重试（指数退避 1s/2s/4s，最多 3 次）
- HTTP 返回非 200 → 记录 metrics，返回空列表
- 业务错误（如 push2 rc ≠ 0）→ 记录 metrics（business_error 类型），不重试
- 解析失败 → 记录日志，跳过该条，不中断整个批次
- 所有异常不抛到调用方，返回空列表

### 2.4 HTTP 客户端

- 使用 M1 创建的 httpx.AsyncClient 全局连接池（从 app.state 获取）
- 同步 HTTP 调用（AkShare、requests）通过 asyncio.to_thread() 包装
- 单次请求超时 15 秒

### 2.5 Stream 发布

- 完成后通过 mq.publish_event 发布到 Stream
- payload 包含：原始数据列表、fetch_source、count、timestamp
- 发布失败（Redis 不可用）→ 记录日志，不抛异常

### 2.6 metrics 记录

每个 fetcher 完成后调用 metrics.record_fetch：
- success=True：HTTP 成功 + 业务成功 + 数据量 > 0
- success=True, business_error=True：HTTP 成功但业务错误（如 push2 rc ≠ 0）
- success=False：网络错误或 HTTP 错误

## 三、fetchers/__init__.py

### 3.1 工具函数

| 函数 | 说明 |
|------|------|
| normalize_code(code) | 补零到 6 位，去空格，全角转半角 |
| safe_float(val, default=None) | 安全浮点转换，处理 None/"-"/NaN |
| retry_fetch(func, max_retries, backoff) | 通用重试装饰器 |
| is_trading_day() | 调用 M1 的 trade_calendar |
| get_http_client() | 从 app.state 获取全局 httpx.AsyncClient |

### 3.2 重试策略

- 最大重试 3 次
- 退避间隔：1 秒 → 2 秒 → 4 秒（指数退避）
- 只对网络错误重试（Timeout、ConnectionError）
- 业务错误不重试（push2 rc ≠ 0、lsjz ErrCode ≠ 0）
- 每次重试记录日志

### 3.3 统一导出

```python
# fetchers/__init__.py 导出
from .realtime import fetch_realtime
from .fundamental import fetch_fundamental
from .historical import fetch_historical
from .info import fetch_info
```

scheduler 和其他模块只需 `from fetchers import fetch_realtime`。

## 四、fetchers/realtime.py

### 4.1 数据源优先级

1. push2 clist（主）：单次请求获取全部 ETF+LOF 实时行情
2. 腾讯 qt（备）：仅覆盖深市（SZ），批量最多 50 个/次

AkShare 交叉验证不在 fetcher 中执行（归 M4 validator 的后台异步验证）。

### 4.2 push2 clist 请求

URL：https://push2.eastmoney.com/api/qt/clist/get

请求参数：
- fs：板块过滤，覆盖沪深 LOF+ETF
- fields：f2(最新价) / f3(涨跌幅) / f5(成交量) / f6(成交额) / f8(换手率) / f10(量比) / f12(代码) / f13(市场) / f14(名称) / f15(涨停价) / f16(跌停价) / f18(昨收) / f20(总市值) / f21(流通市值) / f37(外盘) / f38(内盘) / f204(盘中估值)
- pn=1, pz=2000（单次获取全部）

响应结构：
- rc=0 表示成功
- data.diff 是基金列表数组
- 每个元素的 f12 是代码，f14 是名称

### 4.3 push2 字段码抽查

push2 的字段码（f2/f3 等）是东方财富自定义的，没有官方文档，可能随时变更。

抽查逻辑：
- 取前 5 只基金的 f2（最新价）
- 校验：f2 > 0 且和 f18（昨收）偏差 < 20%
- 5 只中 3 只以上异常 → 大概率字段码变了 → 中断本次采集，告警
- 不写入 Redis，不发布 Stream 事件

### 4.4 部分数据防护

- 记录上次采集的数据量（存储在内存变量中）
- 本次数据量 < 上次的 80% → 判定为数据不完整
- 不覆盖旧数据，保留上一次的有效数据
- 记录告警日志

### 4.5 腾讯 qt 备源

URL：https://qt.gtimg.cn/q=sz160644,sz160645,...

请求方式：
- 拼接代码列表，单次最多 50 个
- 需要多批次请求（1500 只 / 50 = 30 批）
- 仅覆盖深市（SZ 前缀），沪市（SH）无法使用此备源

返回格式：腾讯 qt 返回的是文本格式（非 JSON），需要按分隔符解析。

字段映射（腾讯 qt 的字段格式和 push2 完全不同，M4 normalize 会处理）：
- 腾讯返回的字段较少（价格/成交量/成交额），缺失字段标记为 None

### 4.6 输出格式

fetch_realtime 返回原始数据列表，每个元素保留 push2 的原始字段名（f2/f3/f12 等），加上 fetch_source 和 timestamp。不做字段映射。

## 五、fetchers/fundamental.py

### 5.1 数据源

lsjz API：https://api.fund.eastmoney.com/f10/lsjz

请求参数：
- fundCode：基金代码
- pageIndex=1, pageSize=20
- 返回：单位净值(dwjz)、累计净值(ljjz)、净值日期(fsrq)、申购状态(sgzt)、赎回状态(shzt)、日增长率(rzzl)

### 5.2 并发策略

- Semaphore(5)：最多 5 个并发请求
- 请求间隔 0.3 秒
- 1,500 只基金 / 5 并发 = 300 批 × 0.3 秒 ≈ 1.5 分钟纯等待 + 请求时间 ≈ 3-5 分钟
- 全部跑完后，失败的基金重试一轮

### 5.3 字段解析逻辑

lsjz 返回的字段是中文拼音缩写：
- dwjz → 单位净值
- fsrq → 净值日期（格式：2026-05-29）
- sgzt → 申购状态（中文："开放申购" / "暂停申购" / "限制大额"）
- shzt → 赎回状态（中文："开放赎回" / "暂停赎回"）
- rzzl → 日增长率

sgzt 解析规则：
- 包含"开放" → "open"
- 包含"暂停" → "suspended"
- 包含"限制大额" → "restricted"
- 其他 → "unknown"

shzt 解析规则：
- 包含"开放" → "open"
- 包含"暂停" → "suspended"
- 其他 → "unknown"

### 5.4 QDII 处理

QDII 基金的净值通常 T+1 甚至 T+2 才出：
- fetcher 不做特殊处理，只记录 nav_date
- 消费者入库时判断：nav_date < 今日 → 标记 nav_type = "estimated"
- fetcher 的输出中保留原始 nav_date，让消费者决策

### 5.5 清洗规则

| 条件 | 处理 |
|------|------|
| nav ≤ 0 | 丢弃整条记录 |
| nav_date 无效或为空 | 丢弃整条记录 |
| sgzt 为空 | 设为 "unknown" |
| ErrCode ≠ 0 | 跳过该基金 |
| HTTP 超时 | 重试，仍失败则跳过 |

### 5.6 费率数据

lsjz API 不返回费率数字（申购费率、赎回费率等）。费率数据来自 fundf10（M3 info.py）。lsjz 只返回申赎状态（开放/暂停），不返回费率金额。

## 六、fetchers/historical.py

### 6.1 数据源

- 主源：push2his（东方财富 K 线 API，含成交量和换手率）
- 备源：AkShare fund_etf_hist_em

### 6.2 push2his 请求

URL：https://push2his.eastmoney.com/api/qt/stock/kline/get

请求参数：
- secid：市场前缀 + 代码（深市 0.160644，沪市 1.501225）
- fields2：f51(日期) / f52(开盘) / f53(收盘) / f54(最高) / f55(最低) / f56(成交量) / f57(成交额) / f59(涨跌幅) / f61(换手率)
- klt=101（日线）
- fqt=1（前复权）
- end=20500101（获取全部历史）

多端点降级：
- 先尝试 push2.eastmoney.com（速度快但偶尔不稳定）
- 失败后降级到 push2his.eastmoney.com（更稳定）

### 6.3 AkShare 备源

- asyncio.to_thread(ak.fund_etf_hist_em)（同步库，必须包装）
- 返回 DataFrame，字段名为中文
- M4 normalize 会处理中文字段到英文字段的映射
- AkShare 响应较慢（5-10 秒），但数据完整

### 6.4 底层资产日线

除了基金日线，还需要获取底层资产（股票）的日线：
- 数据来源：从 fund_asset_map 表获取持仓股票代码列表
- A 股：push2his，secid 前缀 0.（深市）或 1.（沪市）
- 港股：push2his，secid 前缀 116.
- 美股：新浪 API 或 AkShare（时差，凌晨单独跑）

首次部署问题：fund_asset_map 为空（info.py 还没爬完持仓），资产日线采集无数据可采集。接受前 5 天资产日线为空，前端做好空数据展示。

### 6.5 并发策略

- Semaphore(5)：最多 5 个并发
- 请求间隔 0.2 秒
- 540 只基金 × 0.2 秒 ≈ 2 分钟 + 请求时间 ≈ 5-10 分钟
- 每只基金一次请求获取全部历史日线

### 6.6 清洗规则

| 条件 | 处理 |
|------|------|
| close ≤ 0 | 丢弃该行（停牌日） |
| volume < 0 | 丢弃该行 |
| trade_date 无效 | 丢弃该行 |
| push2his 返回 rc ≠ 0 或 klines 为空 | 切换到 AkShare 备源 |

## 七、fetchers/info.py

### 7.1 数据源

fundf10 HTML 爬取。需要访问 3 个页面获取完整信息：

| 页面 | URL | 获取内容 |
|------|-----|---------|
| 费率页 | fundf10.eastmoney.com/jjfl_{code}.html | 申购费率、赎回费率、申购限额 |
| 持仓页 | fundf10.eastmoney.com/FundArchivesDatas.aspx?type=jjcc&code={code} | 十大持仓（JSON 格式，通过 AJAX 接口获取） |
| 基本信息页 | fundf10.eastmoney.com/jbgk_{code}.html | 基金类型、跟踪指数、规模、上市日期、赎回到账天数 |

### 7.2 持仓页特殊说明

持仓页不是普通 HTML 页面，是通过 AJAX 接口返回 JSON 数据：
- URL：FundArchivesDatas.aspx?type=jjcc&code={code}
- 返回：JavaScript 变量赋值格式（var apidata={...}）
- 解析：提取 JSON 部分，json.loads 解析
- 比 HTML 解析更稳定

### 7.3 分批策略

- 从 fetch_progress 表读取上次进度（last_code、failed_codes）
- 第一步：重试昨天失败的基金（从 failed_codes 中取，最多 300 只）
- 第二步：取下一批 300 只新基金
- 每只基金完成后更新 fetch_progress（last_code、completed）
- 崩溃后重启可续接（从 last_code 继续）

### 7.4 请求间隔

- 每只基金间隔 2 秒（3 个页面 × 每个页面的请求时间）
- 300 只 × 2 秒 = 600 秒 = 10 分钟/天
- 1,500 只 / 300 只/天 = 5 天完成一轮

### 7.5 清洗规则

| 条件 | 处理 |
|------|------|
| 费率解析失败 | 设为 None，不丢弃整条记录 |
| 赎回费率无数据 | 设为 None，不给默认值（如 1.5%） |
| 申购限额显示"无限额" | 设为 None |
| 申购限额显示"暂停申购" | 设为 None，purchase_status 标记 suspended |
| 持仓为空 | 设为空列表 |
| 基金代码不匹配（HTML 结构变更） | 记录告警，跳过 |

### 7.6 HTML 结构变更检测

fundf10 的 HTML 页面结构随时可能变更（东方财富改版），导致解析失败。

检测逻辑：
- 连续 3 只基金的持仓解析结果都是空列表
- 大概率是 HTML 结构变了，不是真的没有持仓
- 中断本次采集，发送告警到企业微信
- 不继续浪费请求配额

### 7.7 进度持久化

fetch_progress 表字段：
- task_name：任务名（"fund_info"）
- last_code：上次爬到的基金代码
- last_run：上次执行时间
- total：总基金数
- completed：已完成数
- failed_codes：失败的基金代码列表（JSONB 数组）

续接逻辑：
- 启动时查 fetch_progress
- last_code 存在且在当前代码列表中 → 从 last_code 之后继续
- last_code 不在当前代码列表中（基金退市）→ 从头开始
- failed_codes 不为空 → 先重试失败的

## 九、代码列表扫描（scan_codes）

每周一 08:30 由 scheduler 触发。

数据源：push2 clist（和 realtime 用同一个接口，但只需代码和名称）。

逻辑：
- 调用 push2 clist 获取全部 ETF+LOF 代码和名称
- 和 DB 中的 fund_code_list 对比
- 新增的基金 → 插入 fund_code_list
- 名称变更的基金 → 更新 fund_code_list.name
- 不删除退市基金（保留历史数据的完整性）

## 十一、注意事项汇总

### 外部 API 风险

- push2 字段码无官方文档，可能随时变更 → 字段码抽查机制
- fundf10 HTML 结构可能变更 → 连续空值检测 + 告警
- lsjz 可能限流 → Semaphore(5) + 间隔 0.3 秒
- push2his 可能超时 → AkShare 备源
- 腾讯 qt 仅深市 → 沪市基金无法用此备源

### 数据一致性

- 基金名称以 push2 为准，其他来源不入库
- fetch_source 字段透传到 DB，可追溯数据来源
- fetch_batch_id 透传到 DB，可追溯采集批次

### 资源管理

- 使用 M1 的 httpx.AsyncClient 全局连接池，不创建新连接
- AkShare 是同步库，必须用 asyncio.to_thread() 包装
- fundf10 的 requests.get() 也是同步，同样需要包装

### 首次部署

- fund_asset_map 为空 → 资产日线采集无数据 → 接受前 5 天为空
- info.py 分批 5 天完成第一轮 → 前 5 天基础信息/费率/持仓逐步填充
