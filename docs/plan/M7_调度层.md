# M7 调度层 详细设计

## 一、模块总览

| 文件 | 职责 | 依赖 |
|------|------|------|
| scheduler.py | APScheduler 定时任务定义 + 重启补执行 + 错误处理 | fetchers, mq, metrics, trade_calendar, cache |

核心约束：scheduler 只触发 fetcher 和发布 Stream 事件，不直接操作 DB 或 Redis 缓存。数据入库只有一条路径：fetcher → Stream → 消费者。

---

## 二、9 个定时任务

| 任务 | 触发时间 | 调用 | 事件 type | 交易日检查 | 说明 |
|------|---------|------|-----------|-----------|------|
| scan_codes | 周一 08:30 | push2 clist | info | 否（每周固定） | 代码列表扫描 |
| fetch_info | 周一~五 09:00 | fetcher/info | info | 是 | 每天 300 只分批 |
| fetch_realtime | 交易时段每 5 分钟 | fetcher/realtime | realtime | 是 | 内部判断交易日 |
| fetch_nav | 每日 20:00 | fetcher/fundamental | nav | 是 | 普通基金净值+申赎 |
| fetch_kline | 每日 20:30 | fetcher/historical | kline | 是 | 日线 K 线 |
| fetch_nav_qdii | 每日 23:00 | fetcher/fundamental | nav | 是 | QDII 补充净值 |
| daily_save | 每日 23:30 | 无 fetcher，直接发事件 | daily_save | 是 | 触发消费者入库 |
| check_partitions | 每月 1 日 09:00 | ensure_partition | 无 | 否 | 检查下月分区 |
| check_calendar | 每年 12 月 1 日 09:00 | migration | 无 | 否 | 检查下年日历 |

---

## 三、Job 统一配置

| 参数 | 值 | 说明 |
|------|-----|------|
| max_instances | 1 | 同一 job 不并发执行 |
| coalesce | True | 多次触发合并为一次 |
| misfire_grace_time | 300 | 错过触发时间 5 分钟内仍可执行，超过则跳过 |

每个 job 的执行函数必须 try-except 包裹。异常记录 job_log（status=failed, error_msg），不中断 scheduler，不影响其他 job。

## 四、每个任务的详细逻辑

### 4.1 scan_codes（代码列表扫描）

触发时间：每周一 08:30（在 fetch_info 之前 30 分钟）

逻辑：
1. 调用 push2 clist 获取全部 ETF+LOF 代码和名称
2. 和 DB 中的 fund_code_list 对比
3. 新增的基金 → 插入 fund_code_list
4. 名称变更的基金 → 更新 fund_code_list.name
5. 不删除退市基金（保留历史数据完整性）
6. 记录 job_log

时间错开的理由：fetch_info 在 09:00 执行时，代码列表已更新。如果 scan_codes 和 fetch_info 同时触发，fetch_info 可能用旧列表。

### 4.2 fetch_info（持仓+基础信息分批）

触发时间：周一~五 09:00

逻辑：
1. 从 fetch_progress 读取上次进度（last_code、failed_codes）
2. 第一步：重试昨天失败的基金（从 failed_codes 取，最多 300 只）
3. 第二步：取下一批 300 只新基金（从 last_code 之后）
4. 调用 fetcher/info 爬取
5. 每只完成后更新 fetch_progress
6. 记录 job_log（duration_ms, records_in, records_out）

续接逻辑：
- last_code 存在且在当前代码列表中 → 从 last_code 之后继续
- last_code 不在当前代码列表中（基金退市或代码列表更新）→ 从头开始

### 4.3 fetch_realtime（实时行情）

触发时间：交易时段每 5 分钟（9:30-15:00）

逻辑：
1. 检查 is_trading_day()，非交易日 → return
2. 调用 fetcher/realtime
3. fetcher 发布 realtime 事件到 Stream
4. 记录 job_log

触发方式：APScheduler interval trigger，start_date=09:30, end_date=15:00, day_of_week=mon-fri。

注意：interval trigger 的 start_date 和 end_date 只控制触发窗口。job 函数内部仍然需要 is_trading_day() 检查，因为节假日调休时周一~五可能不是交易日。

### 4.4 fetch_nav（普通基金净值）

触发时间：每日 20:00

逻辑：
1. 检查 is_trading_day()，非交易日 → return
2. 调用 fetcher/fundamental（获取所有基金的净值+申赎）
3. fetcher 发布 nav 事件到 Stream
4. 记录 job_log

### 4.5 fetch_kline（日线 K 线）

触发时间：每日 20:30

逻辑：
1. 检查 is_trading_day()
2. 调用 fetcher/historical（基金日线 + 资产日线）
3. fetcher 发布 kline 事件到 Stream
4. 记录 job_log

### 4.6 fetch_nav_qdii（QDII 补充净值）

触发时间：每日 23:00

逻辑：
1. 检查 is_trading_day()
2. 调用 fetcher/fundamental（只获取 QDII 基金的净值）
3. fetcher 发布 nav 事件到 Stream
4. 记录 job_log

为什么要单独跑一次：QDII 基金净值通常延迟（T+1 或 T+2），20:00 的 lsjz 请求可能拿不到今天的净值。23:00 再试一次，如果仍然没有，daily_save 会用昨日净值替代。

### 4.7 daily_save（日终入库）

触发时间：每日 23:30

逻辑：
1. 检查 is_trading_day()
2. 不调用 fetcher，直接通过 mq.publish_event 发布 daily_save 事件
3. 消费者收到事件后执行：合并收盘价+净值+日线 → 计算派生字段 → 分批写 DB → 刷新物化视图
4. 记录 job_log

daily_save 不直接操作 DB 或 Redis。它只是一个触发信号，实际入库逻辑在消费者（M4 pipeline.py）中。

### 4.8 check_partitions（分区检查）

触发时间：每月 1 日 09:00

逻辑：
1. 计算下月日期
2. 调用 ensure_partition（检查下月的 asset_daily 分区是否存在）
3. 不存在 → 创建
4. 记录 job_log

注意：这个任务只检查下月分区。当月分区在 migration.py 启动时和 saver.py 写入前也会检查（三重保障）。

### 4.9 check_calendar（日历检查）

触发时间：每年 12 月 1 日 09:00

逻辑：
1. 查询 trade_calendar 中下一年的交易日数量
2. 不存在或数量异常（不在 200~260 范围）→ 发送告警
3. 告警提示：需要更新下一年的交易日历数据

## 五、重启补执行

进程崩溃重启后，某些任务可能错过了触发时间。

补执行检查逻辑（在 lifespan 启动时执行）：
1. 遍历 scheduler 中注册的所有 job
2. 对每个 job，判断：当前时间 > 今天该任务的触发时间 AND 今天是该任务应该执行的日子
3. 查询 job_log：今天是否有该 job 的 status=success 记录
4. 如果没有 → 立即补执行（modify next_run_time = now + 5 秒）

特殊处理：
- fetch_realtime：如果错过了 14:55 的触发，补执行一次即可（不补跑整个下午的）
- daily_save：如果错过了 23:30，补执行一次
- fetch_info：如果错过了 09:00，补执行一次（继续从上次进度）

## 六、QDII 依赖处理（daily_save 内部）

23:30 daily_save 触发后，消费者的处理逻辑：

1. 从 Redis 读取收盘价（rt:close:{date}）和净值（nav:all）
2. 合并数据时，检查每只基金是否有当日净值
3. 非 QDII 基金无净值 → 跳过（停牌或数据缺失）
4. QDII 基金无当日净值 → 用昨日净值替代，标记 nav_type = "estimated"
5. 先入库非 QDII 基金 + 刷新物化视图
6. QDII 延迟重试：每 5 分钟检查一次，最多 3 次
7. 重试期间如果拿到了当日 QDII 净值 → 追加更新 DB + 再次刷新物化视图
8. 3 次都失败 → QDII 基金保持 estimated 状态

判断是否 QDII：查 fund_info 表的 fund_type 字段。如果 fund_type = "QDII"，归类为 QDII 基金。

## 七、fetch_info 分批逻辑

每天处理一批（300 只），1,500 只 / 300 = 5 天完成一轮。

详细流程：
1. 从 fetch_progress 表读取 task_name="fund_info" 的进度
2. 如果 failed_codes 不为空 → 先取 failed_codes 中的前 300 只进行重试
3. 重试完成后，清除已成功的 failed_codes
4. 如果重试后还有余量（< 300 只已处理）→ 从 last_code 之后取新基金补齐 300 只
5. 调用 fetcher/info 爬取
6. 每只完成后更新 fetch_progress（last_code、completed）
7. 记录 job_log

一周的执行节奏：
- 周一：scan_codes 08:30 → fetch_info 09:00（批次 1：基金 1-300）
- 周二：fetch_info 09:00（批次 2：基金 301-600）
- 周三：fetch_info 09:00（批次 3：基金 601-900）
- 周四：fetch_info 09:00（批次 4：基金 901-1200）
- 周五：fetch_info 09:00（批次 5：基金 1201-1500）
- 下周一：重新开始（或继续失败的）

## 八、错误处理

### 8.1 job 级别

每个 job 的执行函数 try-except 包裹：
- 成功 → 记录 job_log（status=success, duration_ms, records_in, records_out）
- 失败 → 记录 job_log（status=failed, error_msg, detail）
- 不中断 scheduler
- 不影响其他 job

### 8.2 连续失败告警

对每个 job 维护连续失败计数：
- 失败 → 计数 +1
- 成功 → 计数归零
- 计数 ≥ 3 → 调用 metrics.alert() 发送告警

告警去重：同一 job 同一小时内不重复告警。

### 8.3 毒消息防护

Stream 消费者处理事件时，如果同一个事件连续失败 3 次 → 强制 ack，不再重试。记录到 job_log 中，标注为"毒消息丢弃"。

## 九、交易日判断

每个 job 函数的第一行调用 is_trading_day()：
- 返回 True → 继续执行
- 返回 False → 记录日志 "非交易日，跳过"，直接 return

is_trading_day() 从 M1 的 trade_calendar 模块获取：
- 启动时加载当年+明年日历到内存 dict
- 查询是 O(1)，零 DB 开销
- 日历数据不存在 → 返回 False（宁可不采也不乱采）

需要跳过交易日检查的任务：scan_codes（每周固定）、check_partitions（每月固定）、check_calendar（每年固定）。

## 十一、注意事项

### APScheduler 行为

- coalesce=True：如果进程在 14:50 崩溃，14:55 和 15:00 两次触发会被合并为一次
- max_instances=1：如果上一次 fetch_realtime 还没跑完（外部 API 慢），下一次触发被跳过
- misfire_grace_time=300：如果进程在 23:29 崩溃，23:31 重启，23:30 的 daily_save 还在 5 分钟宽限期内，会补执行
- 如果进程在 23:36 重启，daily_save 已超过宽限期，需要靠重启补执行逻辑

### 资源竞争

- fetch_realtime（每 5 分钟）和 fetch_info（09:00）可能同时运行（09:00-09:05 重叠）
- 两者都使用 httpx.AsyncClient 全局连接池，连接池大小 20，足够同时运行
- Semaphore 限制各自的并发数，不会同时发出过多外部请求

### 非交易日处理

- fetch_realtime：interval trigger 在非交易日也会触发（day_of_week=mon-fri 不含节假日），靠 is_trading_day() 跳过
- fetch_nav / fetch_kline：cron trigger 每天触发，靠 is_trading_day() 跳过
- scan_codes / fetch_info：周一~五触发，非交易日也可以跑（采集基础信息不需要交易日数据）

## 十二、验证标准（12 项）

| 测试 | 预期 |
|------|------|
| 交易日 09:30-15:00 | fetch_realtime 每 5 分钟触发 |
| 非交易日 | 交易相关任务跳过（日志记录"非交易日"） |
| 法定假日（周一但是非交易日） | fetch_realtime 跳过 |
| 进程 23:29 崩溃 23:31 重启 | daily_save 补执行（在 misfire_grace_time 内） |
| 进程 23:36 重启 | daily_save 重启补执行逻辑触发 |
| QDII 净值 23:00 未更新 | 23:30 用昨日净值，标记 estimated |
| QDII 净值 23:35 更新 | 追加更新 DB，再次刷新物化视图 |
| QDII 净值 3 次重试仍无 | 保持 estimated 状态 |
| 代码列表更新后 fetch_info | 从新列表的正确位置继续分批 |
| 任务失败 | job_log 记录 status=failed + error_msg |
| 连续 3 次失败 | metrics.alert() 触发告警 |
| fetch_realtime 超过 5 分钟 | 下一次触发被跳过（max_instances=1） |
