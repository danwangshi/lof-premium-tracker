## 二、exceptions.py

### 2.1 异常类层级

```
AppException（基类，继承 HTTPException）
  ├── AuthException          401  鉴权失败
  ├── PermissionException    403  权限不足
  ├── ValidationException    400  参数/输入校验失败
  ├── NotFoundException      404  资源不存在
  ├── RateLimitException     429  请求限流
  └── ServerException        500  服务端内部错误
```

### 2.2 基类属性

| 属性 | 类型 | 说明 |
|------|------|------|
| code | int | 业务错误码（如 40102），5 位数字 |
| message | str | 用户可读的错误描述 |
| detail | str/dict/list/None | 技术细节，仅 DEBUG 模式返回前端 |
| status_code | int | HTTP 状态码，从 code 前 3 位推导（40102 → 401） |
| error_category | str | 错误分类，用于日志过滤和 metrics |

构造函数接收 code、message、detail 三个参数。status_code 由 code // 100 推导。error_category 由 code 的分类决定。

### 2.4 统一响应格式

成功响应：
```
{code: 0, data: {...}, meta: {...}, message: "success"}
```

错误响应：
```
{code: 40102, message: "登录已过期，请重新登录", detail: "exp: ..."}
```

detail 字段：config.DEBUG = true 时返回技术细节，生产环境为 null。

Pydantic 校验错误响应：
```
{code: 40001, message: "参数不合法", detail: [{field: "size", error: "must be <= 100", input: 200}]}
```

### 2.5 全局异常处理器（3 个，按优先级）

| 优先级 | 捕获类型 | 转换逻辑 |
|--------|----------|----------|
| 1 | AppException | 直接提取 code/message/detail，构建统一 JSONResponse |
| 2 | RequestValidationError | 转为 40001，从 Pydantic 错误中提取字段名+错误原因+输入值，构建 detail 数组 |
| 3 | Exception（兜底） | 包装为 50000，完整 traceback 写日志（不返回前端），触发错误现场快照 |

### 2.6 Pydantic 校验错误的 detail 提取逻辑

FastAPI 的 RequestValidationError 包含一个 errors 列表，每个 error 有 loc（位置）、msg（消息）、type（类型）。提取逻辑：遍历 errors 列表，从 loc 中取最后一个元素作为字段名（路径参数和 body 参数的 loc 格式不同），构建 {field, error, input} 数组。

### 2.7 错误码 JSON 导出

提供一个 export_error_codes_json 函数，导出全部错误码定义为 JSON 格式。用途：AI 排查问题时参考、前端错误码映射表、API 文档自动生成。

## 三、auth/middleware.py

### 3.1 中间件执行顺序

FastAPI/Starlette 中间件采用包装模型：先添加的先执行（最外层）。

```
代码顺序: app.add_middleware(AuthMiddleware) → app.add_middleware(CORSMiddleware)
执行顺序: CORSMiddleware（最外层） → AuthMiddleware（内层） → 路由
```

关键：CORSMiddleware 必须在最外层，确保 OPTIONS 预检请求直接返回 CORS header，不走鉴权。错误响应（401/403/500）也能带上 CORS header，否则浏览器收不到错误响应体。

### 3.2 鉴权处理流程

请求进入 AuthMiddleware：

1. 提取 Authorization header
2. header 不存在 → 设置 request.state = {user_id: None, email: None, role: "anonymous"}，放行
3. header 存在但格式错误（不是 "Bearer xxx"）→ 抛 AuthException(40104)
4. 提取 token 部分，调用 jwt.decode：
   - 抛出 ExpiredSignatureError → 抛 AuthException(40102, detail=过期时间)
   - 抛出 InvalidTokenError → 抛 AuthException(40103, detail=错误类型)
   - 解码成功但 payload 缺少 sub 字段 → 抛 AuthException(40105)
   - 解码成功且 sub 存在 → 继续
5. 从 payload 提取信息：
   - user_id = payload["sub"]（Supabase 用户 UUID）
   - email = payload.get("email")
   - role 判断：email 在 config.ADMIN_EMAILS 列表中 → "admin"，否则 → "authenticated"
6. 注入 request.state = {user_id, email, role}
7. 记录请求日志
8. 放行

### 3.3 JWT 验证参数

- 算法：HS256
- 密钥：config.SUPABASE_JWT_SECRET（从 Supabase Dashboard 获取）
- options：verify_aud = False（Supabase 的 aud 字段值不固定，可能是 "authenticated" 或项目 ID）
- 只验证签名和过期时间，不验证 aud

### 3.4 admin 判断逻辑

- config.ADMIN_EMAILS 是逗号分隔的邮箱字符串
- 启动时解析为 set 存储（O(1) 查找）
- 中间件中判断：email in admin_emails_set → role = "admin"
- 零 DB 开销，纯内存判断
- ADMIN_EMAILS 为空时，所有用户 role = "authenticated"，无人是 admin

### 3.5 请求日志

鉴权通过后记录：
- 格式：[AUTH] user={user_id前8位} method={method} path={path} ip={client.host}
- 鉴权失败时：[AUTH] REJECTED code={错误码} ip={client.host}
- 匿名请求：[AUTH] anonymous method={method} path={path}

### 3.6 边界情况处理

| 场景 | 行为 |
|------|------|
| token 过期 | 40102，前端收到后静默刷新 token，用新 token 重试 |
| JWT Secret 轮换 | 40103，所有现有 token 失效，前端用户需重新登录 |
| ADMIN_EMAILS 为空 | 所有用户 role = "authenticated"，admin 端点全部 40301 |
| ADMIN_EMAILS 包含不存在的邮箱 | 无影响，只是该邮箱永远无法匹配 |
| 请求体过大 | Nginx 层拦截（client_max_body_size），不进 middleware |
| OPTIONS 预检请求 | CORSMiddleware 直接返回 200 + CORS headers，不进 AuthMiddleware |

## 四、auth/dependencies.py

### 4.1 三个依赖函数

| 函数 | 逻辑 | 异常 | 使用场景 |
|------|------|------|---------|
| get_user_id | 从 request.state 取 user_id，为 None → 抛 AuthException(40101) | 40101 | 必须登录的路由（基金列表、公式 CRUD） |
| require_admin | 从 request.state 取 role，不是 "admin" → 抛 PermissionException(40301) | 40301 | 管理员路由（监控、操作、日志） |
| get_optional_user | 从 request.state 取 user_id，为 None → 返回 None（不报错） | 无 | 可选登录的路由（基金详情，登录后返回个性化数据） |

### 4.2 两级 admin 权限

admin_read：诊断查询、日志查看、缓存查看、监控状态。用 require_admin 依赖。

admin_write：清缓存、触发刷新、热更新配置、DB vacuum。用 require_admin 依赖 + 额外在 system_service 中校验（可选：配置项控制哪些 admin 邮箱有 write 权限）。

### 4.3 路由使用示例

| 路由 | 依赖 | 效果 |
|------|------|------|
| GET /api/v1/health | 无 | 公开，任何人可访问 |
| GET /api/v1/funds | Depends(get_user_id) | 必须登录 |
| GET /api/v1/funds/:code | Depends(get_optional_user) | 可选登录 |
| POST /api/v1/refresh | Depends(require_admin) | 管理员 only |
| POST /api/formulas | Depends(get_user_id) | 必须登录 |
| GET /api/v1/admin/monitor | Depends(require_admin) | 管理员 only |

## 五、CORS 配置

### 5.1 配置项

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| CORS_ORIGINS | https://jinkuaicha.com | 允许的源，逗号分隔多个。不用 * |
| CORS_ALLOW_METHODS | GET,POST,PUT,DELETE,OPTIONS | 允许的 HTTP 方法 |
| CORS_ALLOW_HEADERS | Authorization,Content-Type | 允许的请求头 |

### 5.2 关键配置

- allow_origins：精确匹配前端域名，不用 *。因为请求带 Authorization header（自定义 header），浏览器要求 origin 精确匹配
- allow_credentials=True：支持 cookie（虽然主要用 Bearer token）
- OPTIONS 预检请求：CORSMiddleware 直接返回 200 + CORS headers，不走 AuthMiddleware

### 5.3 错误响应的 CORS headers

全局异常处理器返回的 JSONResponse 也必须带 CORS headers。FastAPI 的 CORSMiddleware 会在响应上自动添加，包括错误响应。但如果手动构造 JSONResponse 且状态码不在 CORSMiddleware 的允许范围内，需要确认 CORS headers 是否正确添加。

## 六、安全注意事项

### 6.1 密钥安全

- SECRET_KEY 至少 32 字符
- JWT Secret 和代码分离（存 .env，不硬编码）
- ADMIN_EMAILS 不暴露到前端响应中
- .env 文件权限 600

### 6.2 响应安全

- 错误响应不泄露内部信息（生产环境 detail 为 null）
- CORS 不用 *
- OPTIONS 请求不走鉴权
- 不在响应中返回 token
- 不在日志中记录完整 token（只记录前 8 位或 user_id）

### 6.3 防攻击

- JWT 签名验证防止伪造 token
- token 过期机制限制泄漏后的有效期
- admin 判断在 middleware 层拦截，不依赖路由层单独检查
- Pydantic 参数校验防止注入

## 七、API Key 机制（P1，后续扩展）

支持 API Key 鉴权作为 JWT 的补充，适合程序化访问（量化用户脚本）。

请求方式：Authorization: ApiKey jk_xxxxxxxxxxxx

存储：API Key 存在 DB 中，和 user_id 关联，可设置过期时间、每日调用次数上限。

中间件逻辑：先检查 Bearer token，没有则检查 ApiKey，都没有则设为 anonymous。

用户在个人设置页面生成/管理 API Key。

## 九、验证标准（14 项）

| 测试 | 预期 |
|------|------|
| 无 token 请求公开路由 | 正常响应，user_id 为 None |
| 无 token 请求需登录路由 | code=40101, message="请先登录" |
| 有效 token 请求 | 正常响应，user_id 正确注入 |
| 过期 token | code=40102, message="登录已过期" |
| 伪造 token | code=40103, message="无效的登录凭证" |
| 格式错误 header | code=40104, message="请求凭证格式不正确" |
| payload 缺 sub | code=40105, message="登录信息不完整" |
| 普通用户访问管理员路由 | code=40301, message="需要管理员权限" |
| 用户访问不属于自己的公式 | code=40403（不泄露"存在但不属于你"） |
| 参数不合法 | code=40001 + detail 数组含字段名和错误原因 |
| CORS 预检请求 | 200 + Access-Control-* headers |
| 跨域带 Authorization | 正常通过 |
| 401/403 错误响应 | 带 CORS header（浏览器能读到响应体） |
| DEBUG=false 时 detail | 为 null，不泄露技术细节 |
