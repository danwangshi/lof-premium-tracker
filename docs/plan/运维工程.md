# 运维工程 详细设计

## 一、部署

### 1.1 deploy.sh（一键部署）

#### 前置检查（自动执行，任一失败则中止）

| 检查项 | 命令 | 通过条件 | 失败处理 |
|--------|------|----------|----------|
| 交易时段 | `date +%H%M` | 不在 0925~1505 之间 | 提示确认，默认拒绝 |
| 磁盘空间 | `df / --output=avail` | 可用 > 1GB | 中止，提示清理 |
| DB 连接 | `pg_isready` | 返回 "accepting connections" | 中止 |
| Redis 连接 | `redis-cli ping` | 返回 PONG | 提示确认，可降级部署 |
| .env 文件 | `test -f .env` | 存在 | 中止 |
| Git 状态 | `git status --porcelain` | 干净（无未提交修改） | 中止，提示先 commit |

#### 执行步骤

```
步骤1: 备份当前版本
  命令: cp .env .env.bak && git rev-parse HEAD > .last_deploy
  预期: .last_deploy 写入当前 commit hash
  用途: 回滚时读取

步骤2: 拉取代码
  命令: cd /opt/jinkuaicha && git pull origin main
  预期: "Already up to date." 或更新文件列表
  失败: 网络不通 → 检查 DNS，不继续

步骤3: 安装依赖
  命令: source venv/bin/activate && pip install -r requirements.txt
  预期: "Successfully installed ..." 或 "Requirement already satisfied"
  失败: 某个包安装失败 → 检查 PyPI 可达性，不继续

步骤4: 数据库迁移
  命令: python migration.py
  预期: "Migration complete: N tables, N partitions"
  失败: SQL 执行错误 → 自动回滚迁移，不继续

步骤5: 重启服务
  命令: sudo systemctl restart jinkuaicha
  预期: 无输出（systemctl 静默成功）

步骤6: 健康检查（连续 3 次，间隔 2 秒）
  命令: for i in 1 2 3; do curl -sf http://127.0.0.1:8000/api/v1/health && break || sleep 2; done
  预期: {"status":"ok","database":"connected","redis":"connected"}
  失败: 3 次全部失败 → 触发回滚

步骤7: 记录变更
  命令: git log --oneline $(cat .last_deploy)..HEAD >> CHANGELOG.md
  自动追加本次部署的 commit 到 CHANGELOG.md
```

#### 失败回滚（自动执行）

```
1. git checkout $(cat .last_deploy)
2. pip install -r requirements.txt
3. sudo systemctl restart jinkuaicha
4. curl health check 验证
5. 输出错误日志最后 20 行: journalctl -u jinkuaicha --since "1 min ago" | tail -20
6. 发送告警到企业微信: "部署失败，已回滚到 $(cat .last_deploy)"
```

### 1.2 rollback.sh（一键回滚）

```
步骤1: 确认 .last_deploy 存在
  命令: test -f .last_deploy || echo "无部署记录，无法回滚" && exit 1

步骤2: 读取上一个版本的 commit hash
  命令: cat .last_deploy → 输出如 a3f2b1c

步骤3: 切换代码
  命令: git checkout $(cat .last_deploy)

步骤4: 安装依赖
  命令: pip install -r requirements.txt

步骤5: 重启 + 验证
  命令: sudo systemctl restart jinkuaicha && curl -sf http://127.0.0.1:8000/api/v1/health

步骤6: 确认回滚成功
  预期: health check 返回 ok，输出 "回滚成功: $(cat .last_deploy)"
```

## 二、Nginx 配置

```nginx
server {
    listen 80;
    server_name jinkuaicha.com;

    # --- SSL（certbot 自动配置）---
    # listen 443 ssl;
    # ssl_certificate /etc/letsencrypt/live/jinkuaicha.com/fullchain.pem;
    # ssl_certificate_key /etc/letsencrypt/live/jinkuaicha.com/privkey.pem;

    # --- gzip ---
    gzip on;
    gzip_min_length 256;
    gzip_types application/json text/css application/javascript;
    gzip_vary on;

    # --- 安全头 ---
    add_header X-Content-Type-Options nosniff;
    add_header X-Frame-Options DENY;
    add_header X-XSS-Protection "1; mode=block";

    # --- 请求 ID（全链路追踪）---
    proxy_set_header X-Request-Id $request_id;

    # --- 通用反代设置 ---
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    # --- SSE 长连接 ---
    location /api/v1/stream/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_read_timeout 3600s;
        proxy_buffering off;           # SSE 必须关闭缓冲
        proxy_cache off;
        chunked_transfer_encoding off;
    }

    # --- API 反代 ---
    location /api/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_read_timeout 30s;
        client_max_body_size 1m;
    }

    # --- 管理后台静态文件 ---
    location /admin/ {
        alias /var/www/jinkuaicha/admin/;
        try_files $uri $uri/ /admin/index.html;
        # 管理后台限制 IP（可选）
        # allow 你的IP;
        # deny all;
    }

    # --- 前端静态文件 ---
    location / {
        root /var/www/jinkuaicha;
        try_files $uri $uri/ /index.html;
        # 静态资源长缓存
        location ~* \.(js|css|png|jpg|ico|svg)$ {
            expires 7d;
            add_header Cache-Control "public, immutable";
        }
    }

    # --- 健康检查（Nginx 直接返回，不经过 FastAPI）---
    location = /nginx-health {
        return 200 '{"status":"ok","service":"nginx"}';
        add_header Content-Type application/json;
    }
}
```

## 六、数据库备份

### 6.1 每日自动备份（cron）

```bash
# /etc/cron.d/jinkuaicha-backup
# 每日凌晨 3:00 执行
0 3 * * * deploy /opt/jinkuaicha/scripts/backup.sh >> /var/log/jinkuaicha/backup.log 2>&1
```

**backup.sh 逻辑：**
```
步骤1: 生成备份文件名
  文件名: jinkuaicha_$(date +%Y%m%d_%H%M%S).dump.gz

步骤2: pg_dump 压缩
  命令: pg_dump -Fc jinkuaicha | gzip > /tmp/$FILENAME

步骤3: 上传到阿里云 OSS
  命令: ossutil cp /tmp/$FILENAME oss://jinkuaicha-backup/$FILENAME

步骤4: 清理本地临时文件
  命令: rm /tmp/$FILENAME

步骤5: 删除 OSS 上 7 天前的备份
  命令: ossutil rm oss://jinkuaicha-backup/ --include "jinkuaicha_*.dump.gz" --exclude "jinkuaicha_$(date +%Y%m%d)*" -r

步骤6: 记录备份结果到 job_log 表
  成功: INSERT INTO job_log (job_name, status, detail) VALUES ('backup', 'success', ...)
  失败: INSERT INTO job_log (job_name, status, error_msg) VALUES ('backup', 'failed', ...)
  失败时发送告警到企业微信
```

### 6.2 每周备份验证（cron）

```bash
# /etc/cron.d/jinkuaicha-backup-verify
# 每周日凌晨 4:00 执行
0 4 * * 0 deploy /opt/jinkuaicha/scripts/verify_backup.sh >> /var/log/jinkuaicha/backup.log 2>&1
```

**verify_backup.sh 逻辑：**
```
步骤1: 下载最新备份
  命令: ossutil cp oss://jinkuaicha-backup/latest.dump.gz /tmp/verify.dump.gz

步骤2: 创建临时数据库
  命令: sudo -u postgres createdb jinkuaicha_verify

步骤3: 恢复到临时库
  命令: gunzip -c /tmp/verify.dump.gz | pg_restore -d jinkuaicha_verify

步骤4: 对比行数
  命令:
    prod_count=$(sudo -u postgres psql -t jinkuaicha -c "SELECT count(*) FROM fund_daily;")
    verify_count=$(sudo -u postgres psql -t jinkuaicha_verify -c "SELECT count(*) FROM fund_daily;")
  预期: 两个数字一致

步骤5: 清理
  命令: sudo -u postgres dropdb jinkuaicha_verify && rm /tmp/verify.dump.gz

步骤6: 记录结果
  一致 → job_log 记录 success
  不一致 → 告警"备份验证失败，行数不一致: prod=$prod_count verify=$verify_count"
```

### 6.3 数据库恢复步骤（紧急）

```
前提: OSS 中有可用备份

步骤1: 停止应用
  sudo systemctl stop jinkuaicha

步骤2: 确认 PostgreSQL 运行
  sudo systemctl status postgresql

步骤3: 备份当前（损坏的）数据库（以防万一）
  sudo -u postgres pg_dump -Fc jinkuaicha > /tmp/jinkuaicha_corrupted.dump

步骤4: 删除并重建数据库
  sudo -u postgres dropdb jinkuaicha
  sudo -u postgres createdb jinkuaicha

步骤5: 列出可用备份，选择恢复哪个
  ossutil ls oss://jinkuaicha-backup/ → 选择日期最近的

步骤6: 下载并恢复
  ossutil cp oss://jinkuaicha-backup/jinkuaicha_20260529.dump.gz /tmp/
  gunzip /tmp/jinkuaicha_20260529.dump.gz
  pg_restore -d jinkuaicha /tmp/jinkuaicha_20260529.dump

步骤7: 验证
  sudo -u postgres psql jinkuaicha -c "SELECT count(*) FROM fund_daily;"
  → 预期: 37万+ 行

步骤8: 重启应用
  sudo systemctl start jinkuaicha
  curl -sf http://127.0.0.1:8000/api/v1/health

步骤9: 清理
  rm /tmp/jinkuaicha_*.dump

RTO: 15 分钟
RPO: 最多丢失当天数据（凌晨备份后到故障前的数据）
```

## 七、定时清理

### weekly_cleanup.sh

```bash
#!/bin/bash
# /opt/jinkuaicha/scripts/weekly_cleanup.sh
# 每周日凌晨 5:00 执行（cron）

set -e
LOG="/var/log/jinkuaicha/cleanup.log"
echo "=== Cleanup started: $(date) ===" >> $LOG

# 1. 删除 7 天前的日志
find /var/log/jinkuaicha -name "*.log" -mtime +7 -delete
echo "Old logs cleaned" >> $LOG

# 2. PostgreSQL VACUUM
sudo -u postgres psql jinkuaicha -c "VACUUM ANALYZE fund_daily;"
sudo -u postgres psql jinkuaicha -c "VACUUM ANALYZE asset_daily;"
echo "VACUUM completed" >> $LOG

# 3. 磁盘检查
DISK_USAGE=$(df / --output=pcent | tail -1 | tr -d '% ')
if [ "$DISK_USAGE" -gt 95 ]; then
    echo "CRITICAL: Disk usage ${DISK_USAGE}%" >> $LOG
    curl -s -X POST "$ALERT_WEBHOOK_URL" -d "{\"msgtype\":\"text\",\"text\":{\"content\":\"[P0] 磁盘使用率 ${DISK_USAGE}%\"}}"
elif [ "$DISK_USAGE" -gt 85 ]; then
    echo "WARNING: Disk usage ${DISK_USAGE}%" >> $LOG
    curl -s -X POST "$ALERT_WEBHOOK_URL" -d "{\"msgtype\":\"text\",\"text\":{\"content\":\"[P1] 磁盘使用率 ${DISK_USAGE}%\"}}"
fi

# 4. Redis 内存检查
REDIS_MEM=$(redis-cli info memory | grep used_memory_human | cut -d: -f2 | tr -d '[:space:]')
echo "Redis memory: $REDIS_MEM" >> $LOG

# 5. asset_daily 旧分区检查
OLD_PARTITIONS=$(sudo -u postgres psql -t jinkuaicha -c "
  SELECT tablename FROM pg_tables
  WHERE tablename LIKE 'asset_daily_%'
  AND tablename < 'asset_daily_$(date -d "12 months ago" +%Y%m)';
")
if [ -n "$OLD_PARTITIONS" ]; then
    echo "Old partitions found (can be archived): $OLD_PARTITIONS" >> $LOG
fi

echo "=== Cleanup finished: $(date) ===" >> $LOG
```

## 八、告警规则

| 告警项 | 条件 | 级别 | 通知 | 处理方式 |
|--------|------|------|------|----------|
| 服务不可用 | /health 连续 3 次非 200 | P0 | 企业微信 | RUNBOOK "服务无法访问" |
| 磁盘 > 85% | df 输出 | P1 | 企业微信 | 执行 weekly_cleanup.sh |
| 磁盘 > 95% | df 输出 | P0 | 企业微信 | 立即清理日志 + VACUUM |
| Redis 内存 > 90% | redis-cli info memory | P1 | 企业微信 | 检查 key 数量和 TTL |
| 采集成功率 < 80% | metrics.fetch_success_rate | P1 | 企业微信 | 检查外部 API，切换备源 |
| DB 连接池满 | pg_stat_activity count > 18 | P1 | 企业微信 | kill 长查询: `SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state = 'active' AND query_start < now() - interval '30s';` |
| 备份失败 | backup.sh 返回非 0 | P1 | 企业微信 | 手动执行 pg_dump |
| 慢查询 > 2s | pg log | P2 | 企业微信 | 检查索引，EXPLAIN ANALYZE |
| SSL 证书 < 30 天到期 | certbot 检查 | P2 | 企业微信 | `sudo certbot renew` |

## 十四、紧急处理手册（RUNBOOK.md）

### 服务无法访问

```
1. systemctl status jinkuaicha
   → active (running) → 服务在运行，问题可能在 Nginx
   → inactive (dead) → 服务挂了，继续步骤 2

2. journalctl -u jinkuaicha --since "10 min ago" --no-pager
   → 查看最近日志，找错误原因

3. ss -tlnp | grep 8000
   → 有输出 → 端口在监听，服务正常
   → 无输出 → 服务没启动

4. sudo systemctl restart jinkuaicha

5. curl -sf http://127.0.0.1:8000/api/v1/health
   → {"status":"ok"} → 恢复
   → 超时或错误 → 检查 PostgreSQL 和 Redis 是否正常

6. 如果重启后仍然失败:
   sudo journalctl -u jinkuaicha --since "1 min ago" | tail -20
   → 查看启动错误，常见: DB 连接失败、端口被占用、配置错误
```

### 数据库连接失败

```
1. sudo systemctl status postgresql
   → active → PG 在运行
   → inactive → sudo systemctl start postgresql

2. sudo -u postgres psql -c "SELECT count(*) FROM pg_stat_activity;"
   → 预期 < 20（max_connections）
   → 如果满 → kill 长查询:
     sudo -u postgres psql -c "SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state = 'active' AND query_start < now() - interval '30s';"

3. df -h
   → 磁盘满会导致 PG 无法写入 → 清理磁盘

4. sudo systemctl restart postgresql
   → 等待 5 秒
   → sudo systemctl restart jinkuaicha
```

### Redis 不可用

```
1. sudo systemctl status redis
   → inactive → sudo systemctl start redis

2. redis-cli info memory | grep used_memory_human
   → 接近 128MB → 可能触发了 LRU 淘汰
   → redis-cli DBSIZE → 检查 key 数量

3. sudo systemctl restart redis
   → 重启后缓存全部清空，API 会短暂降级（返回 DB 数据）
   → 下次采集任务运行后缓存自动恢复
```

### 磁盘满了

```
1. df -h → 确认使用率

2. du -sh /var/log/* /var/lib/postgresql/* /var/lib/redis/* | sort -rh | head -10
   → 找到占用最大的目录

3. 清理:
   find /var/log/jinkuaicha -name "*.log" -mtime +1 -delete
   sudo -u postgres psql jinkuaicha -c "VACUUM FULL fund_daily;"
   sudo -u postgres psql jinkuaicha -c "VACUUM FULL asset_daily;"

4. 如果仍然满:
   检查 asset_daily 旧分区 → 可以 DROP 超过 12 个月的分区
   sudo -u postgres psql jinkuaicha -c "DROP TABLE IF EXISTS asset_daily_202505;"
```

### 23:30 入库失败

```
1. 检查执行记录:
   sudo -u postgres psql jinkuaicha -c "SELECT * FROM job_log WHERE job_name = 'daily_save' ORDER BY id DESC LIMIT 5;"

2. 检查 QDII 净值:
   sudo -u postgres psql jinkuaicha -c "SELECT code, nav, nav_date FROM fund_daily WHERE nav_type = 'estimated' AND trade_date = CURRENT_DATE;"

3. 手动触发入库:
   curl -X POST http://127.0.0.1:8000/api/v1/admin/ops/refresh \
     -H "Authorization: Bearer $ADMIN_TOKEN" \
     -d '{"type": "daily_save", "sync": true}'

4. 检查结果:
   curl http://127.0.0.1:8000/api/v1/admin/diagnose/fetcher
   → 确认 daily_save 状态为 success
```

### 外部 API 全部超时

```
1. 检查网络:
   ping push2.eastmoney.com
   curl -sf --max-time 5 "https://push2.eastmoney.com/api/qt/clist/get" || echo "超时"

2. 如果是东方财富限流:
   → 等待 30 分钟后自动恢复
   → 减少采集频率: 通过 admin API 修改 FETCH_INTERVAL=600

3. 如果是服务器 DNS 问题:
   cat /etc/resolv.conf
   → 尝试更换 DNS: nameserver 223.5.5.5 (阿里 DNS)
```
