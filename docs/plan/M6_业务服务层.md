## 二、hub/service.py（编排器）

### 2.1 职责（只做三件事）

| 职责 | 示例 | 说明 |
|------|------|------|
| 调一个 Service 完成完整请求 | get_fund_list → fund_service.get_fund_list() | 不拆分步骤，一次调用返回完整结果 |
| 跨 Service 协调 | eval_formula_group → formula_service 取公式定义 + data_service 取基金数据 → batch_evaluate | 需要多个 Service 协作时，Hub 负责编排 |
| 缓存失效触发 | create_formula → 删除该用户公式缓存 | 写操作后清理相关缓存 |

### 2.2 Hub 不做的事

- 不直接查 DB（交给 Service）
- 不直接读写 Redis（交给 Service 或 cache.py）
- 不做数据合并（交给 fund_service 内部）
- 不做降级逻辑（交给 Service 内部）
- 不做业务计算（交给 formula_engine / calculator）

### 2.3 缓存失效规则

| 操作 | 清除的缓存 key |
|------|---------------|
| create/update/delete formula | 该用户的公式列表缓存 |
| create/update/delete formula group | 该用户的公式组缓存 |
| trigger_refresh(type=realtime) | rt:all |
| trigger_refresh(type=nav) | nav:all |
| 日终入库完成 | rt:all + snapshot:fund_list |
| 物化视图刷新完成 | snapshot:fund_list |

| merge_fee(fund_detail) | 合并费率到基金详情 | Redis fee:{code} |

### 3.2 缓存击穿保护

场景：Redis rt:all 过期瞬间，50 个用户同时请求基金列表。

处理流程：
1. 读 Redis rt:all → 命中 → 直接返回
2. 未命中 → 尝试获取 Redis 锁（SET NX lock:cache:rt:all EX 3）
3. 拿到锁的请求 → 查 DB + 写 Redis 缓存 + 释放锁 → 返回
4. 没拿到锁的请求 → 轮询等待缓存（间隔 50ms，最多等 2 秒）
5. 等到缓存 → 返回
6. 超时未等到 → 返回上一次的缓存值（即使已过期，用 stale 数据兜底）

锁的 key：lock:cache:{实际缓存 key}，TTL 3 秒（防止死锁）。

### 3.3 Redis 降级

Redis 完全不可用时：
- rt:all 读取失败 → 只返回 DB 物化视图数据
- 响应 meta 中 realtime_available = false
- 前端据此隐藏实时溢价率列，显示"实时数据暂不可用"提示
- 不返回过时的实时数据（误导比没有更危险）

### 3.5 筛选增强

| 参数 | 说明 | 数据库字段 |
|------|------|-----------|
| premium_min / premium_max | 溢价率范围 | fund_daily.premium_rate |
| amount_min / amount_max | 成交额范围 | fund_daily.amount |

| purchase_status | 申购状态（open/suspended） | fund_fee.purchase_status |

| filter | "watchlist" → 只返回自选基金 | user_watchlist |
| search | 代码或名称模糊搜索 | fund_daily.code / name |

### 3.6 排序白名单

premium_rate / close / amount / turnover_rate / change_pct / nav / volume / float_share / aum / code / name

排序字段不在白名单中 → 使用默认排序（amount DESC）。

### 3.7 非交易时段处理

fund_service 组装响应时判断当前是否在交易时段（调用 trade_calendar）：
- 交易时段：realtime_price 有值，data_type = "realtime"
- 非交易时段：realtime_price 设为 null，data_type = "closing"

### 3.8 分页排序防重复

排序字段可能有 NULL 值或重复值。ORDER BY 需要加 tie-breaker：
ORDER BY {sort} DESC NULLS LAST, code ASC

NULLS LAST：NULL 值排在最后（不会因为 NULL 在第一页和第二页之间跳动）。
code ASC：相同排序值时按代码排序，保证确定性顺序。

### 6.2 乐观锁

user_formulas 表有 version 字段（整数，初始值 1，每次 UPDATE +1）。

PUT 请求带 If-Match header（值为当前 version）。后端执行 UPDATE 时 WHERE 条件包含 version，如果 version 不匹配（被其他请求改了）→ 返回 409 Conflict + 当前最新 version 和数据。

### 6.3 并发限制（P1 compute）

- 全局 asyncio.Semaphore(1)：同时只执行 1 个 compute 请求
- 单次请求基金上限 200 只，超过拒绝（40005）
- 超时 10 秒，强制返回已完成的部分结果

### 6.4 删除公式检查

删除公式前，检查是否有公式组引用了该公式：
查询 user_formula_groups 表，WHERE formula_ids @> [formula_id]（JSONB 包含查询）。
有引用 → 拒绝删除，返回"该公式被 N 个公式组使用"。
无引用 → 允许删除。

### 6.5 公式名称唯一性

同一用户内公式名称唯一。不同用户可以重名。

校验：SELECT COUNT(*) FROM user_formulas WHERE user_id = ? AND name = ?
重复 → 返回 40001"名称已存在"。

## 七、services/alert_service.py

### 7.1 函数清单

| 函数 | 说明 |
|------|------|
| list_alerts(user_id) | 列出用户的预警规则 |
| create_alert(user_id, data) | 创建预警 |
| delete_alert(user_id, alert_id) | 删除预警 |
| toggle_alert(user_id, alert_id, active) | 启用/禁用预警 |
| check_alerts(trade_data) | 消费者调用：检查所有活跃预警是否触发 |

### 7.2 预警条件格式

存储为 JSONB：
```
{
  "op": "and",
  "conditions": [
    {"field": "premium_rate", "op": ">", "value": 5},
    {"field": "turnover_rate", "op": ">", "value": 3}
  ]
}
```

支持的操作符：and / or
支持的比较符：> / < / >= / <= / == / !=

### 7.3 预警触发逻辑

消费者每次处理 realtime 事件后，调用 check_alerts：
1. 读取所有 is_active = true 的预警规则
2. 对每条规则，从实时数据中取对应基金的数据
3. 按 condition JSON 求值
4. 条件满足 → 标记 is_triggered = true，触发通知（web_push / email / webhook）
5. 条件不再满足 → 重置 is_triggered = false（下次可再次触发）

### 8.2 物化视图刷新锁

asyncio.Lock 全局锁。刷新前获取锁，获取不到 → 返回"刷新进行中"。

刷新使用 REFRESH MATERIALIZED VIEW CONCURRENTLY（不锁读，需要物化视图上有 UNIQUE INDEX）。

### 8.3 操作防重

admin_write 端点执行前，检查 Redis 中 last_admin_op:{action} 时间戳。同一 action 5 秒内重复调用 → 返回"操作刚执行过，请稍后重试"。

### 8.4 审计日志

每个 admin_write 操作执行后，写 admin_audit_log 表：operator / action / target / params / result / created_at。

admin_read 端点不记录审计日志（查询不算危险操作）。

### 8.5 触发刷新行为

| type | 行为 | 响应 |
|------|------|------|
| realtime | 同步调 fetcher/realtime | 立即返回结果 |
| nav | 同步调 fetcher/fundamental | 立即返回结果 |
| kline | APScheduler 立即调度 job | 返回 task_id |
| all | APScheduler 立即调度全量 job | 返回 task_id |

### 8.6 自愈规则

| 条件 | 检查方式 | 动作 |
|------|----------|------|
| Redis 连接失败 | is_redis_available() | 重启 Redis → 5 秒后检查 → 仍失败告警 |
| 物化视图 >24 小时未刷新 | job_log 最新 daily_save 时间 | 触发刷新 → 失败告警 |
| 采集成功率 <80% | metrics.fetch_status | 切换备源 → 告警 |
| 磁盘 >85% | shutil.disk_usage | 清理日志 → 告警 |
| DB 连接池满 | pg_stat_activity count | kill 长时间查询 → 告警 |

## 九、services/sse_service.py

### 9.1 核心机制

后端对比本次和上次的数据差异，只推送变化的基金和字段。

推送格式：
```
data: {"type":"realtime","ts":"...","changes":[
  {"code":"160644","fields":{"realtime_price":2.35,"change_pct":-1.2}},
  {"code":"510300","fields":{"realtime_price":4.16,"amount":123456789}}
]}
```

单条消息 2-5KB（vs 全量 50-80KB）。

### 9.2 心跳

每 30 秒发送心跳：`data: {"type":"heartbeat","ts":"..."}`

防止 Nginx/Cloudflare 超时断连。Nginx 对 SSE 端点单独配置 proxy_read_timeout 3600s。

### 9.3 并发限制

最多 200 个 SSE 连接。超过 → 返回 503 Service Unavailable。

### 9.4 连接生命周期

- asyncio.wait_for(send_event(), timeout=35) 包裹每个连接的发送循环
- 写入失败（BrokenPipeError / ConnectionResetError）→ 捕获异常
- finally 块中清理资源（释放计数器、清理订阅）
- 客户端关闭浏览器 → 服务端检测到写入失败 → 自动清理

### 9.5 非交易时段

非交易时段不推送数据。SSE 连接保持但不发数据（只发心跳）。次日交易时段开始时恢复推送。
